## Report 1 ‚Äî mlx-swift-lm primitives for batching / KV cache / multi-model

### 1. Continuous batching

**Support today:** NO. `TokenIterator`
(`Libraries/MLXLMCommon/Evaluate.swift`) is strictly single-request: the
model is called on one prompt, a single `[KVCache]` array is threaded
through, output shape `[B=1, seq, vocab]`. No `BatchGenerator`, no
batched `TokenIterator`, no `BatchKVCache` in the Swift tree.

**What Apple provides:** `asyncEval()` plus forward-value pipelining
inside `TokenIterator` keeps GPU busy across decode ticks for ONE
stream. `MLXFast.scaledDotProductAttention` already accepts an
arbitrary leading batch dim, so the attention kernel itself is
batch-ready. `KVCacheSimple.update` reads `keys.dim(0)` as `B`, so
caches can hold B>1 ‚Äî but all Swift-side plumbing hands you B=1.

**What we'd need:**
- `BatchKVCache` / `BatchRotatingKVCache` pair (mirror Python PR #941/#1101)
- `BatchTokenIterator` with pad/left-align, joint prefill, per-sequence
  finished-flag mask, per-sequence sampler
- Request admission + padding (continuous batching admits new requests
  mid-loop; `ArraysCache.filter(batchIndices:)` already exists at
  line 1125 of `KVCache.swift`)
- Per-sequence stop-token detection, stream demuxing

**Risk:** MEDIUM-HIGH. All 85 model architectures call
`createAttentionMask(h:cache:)` with a single cache ‚Äî each has been
written for B=1. Adding batched path means auditing or wrapping every
model. 2‚Äì3 weeks for a Llama/Qwen-only MVP.

**Upstream state:**
- Python mlx-lm has shipped batching. Open PRs:
  - [#941](https://github.com/ml-explore/mlx-lm/pull/941) ‚Äî KV quant + continuous batching
  - [#1101](https://github.com/ml-explore/mlx-lm/pull/1101) ‚Äî memory-aware `BatchQuantizedKVCache`
  - [#1129](https://github.com/ml-explore/mlx-lm/pull/1129) ‚Äî batch-size race fix
  - [#873](https://github.com/ml-explore/mlx-lm/pull/873) ‚Äî per-sequence trim
  - [#1090](https://github.com/ml-explore/mlx-lm/pull/1090) ‚Äî thread-local stream
  - [#548](https://github.com/ml-explore/mlx-lm/issues/548) ‚Äî persistent batch KV cache (open)
- `mlx-swift-examples` has nothing comparable open; PR #170 was SD, not LLM.

### 2. Tiered KV cache (hot RAM + cold SSD)

**Support today:** YES ‚Äî foundation fully shipped. `KVCache.swift`:
- `savePromptCache(url:cache:metadata:)` ‚Äî writes safetensors,
  Python-compatible flattened format
- `loadPromptCache(url:) -> ([KVCache], [String: String])` ‚Äî restores
  via `restoreFromMetaState` for every cache subclass
  (`KVCacheSimple`, `RotatingKVCache`, `QuantizedKVCache`,
  `ChunkedKVCache`, `MambaCache`, `ArraysCache`, `CacheList`)
- Per-cache `state: [MLXArray]` + `metaState: [String]` round-trip
- `trimPromptCache(_:numTokens:)` + `canTrimPromptCache(_:)` ‚Äî partial
  prefix reuse

**What we build:**
- Prefix-hash index `(modelID, tokenHash[:n]) ‚Üí safetensors URL`, LRU on disk
- Longest-common-prefix matcher; trim cache; prefill only the delta
- Hot tier: last K caches in a dict. Cold: on-disk. Background promote/demote
- For quantized caches, optional `toQuantized()` on demote, `toUnquantized()` on promote
  (NB: `RotatingKVCache.toQuantized()` fatal-errors at line 715 ‚Äî do not
  quantize sliding-window caches)

**Risk:** LOW. Pure Swift plumbing over APIs that already exist and are
tested (`KVCacheTests.swift`).

**Upstream state:** mlx-lm PR [#982](https://github.com/ml-explore/mlx-lm/pull/982)
(server prompt-cache warmup) + [#1118](https://github.com/ml-explore/mlx-lm/pull/1118)
(honor `--prompt-cache-bytes`) confirm Apple's direction. Nothing
blocking on the Swift side.

### 3. Multi-model load + auto-swap

**Support today:** YES. `ModelContainer` is `Sendable`, wraps
`SerialAccessContainer<ModelContext>`, multiple instances explicitly
supported.

**What Apple provides:** `Memory.memoryLimit`, `Memory.cacheLimit`
(`mlx-swift/Source/MLX/Memory.swift`). `WiredMemoryUtils` +
`WiredBudgetPolicy` / `WiredSumPolicy`
(`MLXLMCommon/WiredMemoryUtils.swift`) already coordinate across
concurrent `container.generate(...)` calls.

**What we build:** LRU `ModelRegistry` with loader lock + memory-pressure
watcher + request routing by `model:` field + serial weight loading.

**Gotcha:** On M-series unified memory, evicting a container doesn't
return memory to the OS until you call `MLX.GPU.clearCache()` (or
cycle `Memory.cacheLimit`).

**Risk:** LOW. All APIs exist. Not even an open upstream issue.

### Recommendation

Ship in v0.4.0 (Swift-only): tiered prompt cache + multi-model
registry. Both additive over stable APIs, high user-visible win for
coding assistants re-sending history.

Defer continuous batching: requires model-level changes; upstream
Swift hasn't landed it. Wait for Apple port of Python
BatchKVCache (#941/#1101), or carry a fork with Llama/Qwen-only
batched path. Revisit v0.5+.

## Report 2 ‚Äî oMLX implementation study

Repo: `github.com/jundot/omlx` (Apache-2.0, Python, on top of
`mlx-lm`, 10.6k stars).

### 1. Continuous batching

- **Files:** `omlx/scheduler.py` (~4,300 LOC) ‚Äî central orchestrator.
  Monkey-patches `mlx_lm.generate.GenerationBatch._step` (119‚Äì150)
  for grammar + mRoPE.
- **Classes:** `Scheduler`, `SchedulerConfig` (line 297),
  `SchedulingPolicy` (FCFS / PRIORITY). Reuses mlx-lm's
  `BatchGenerator`, `GenerationBatch`, `PromptProcessingBatch`,
  `SequenceStateMachine`.
- **Algorithm:** vLLM-style waiting-queue + running-set. Chunked
  prefill (`prefill_step_size=2048`), admission bounded by
  `max_num_seqs=256` + `max_num_batched_tokens=8192`. Completion
  batch=32. No custom paged-attention ‚Äî relies on mlx-lm's
  `BatchGenerator`.
- **Portable concepts for macMLX:** FCFS waiting-queue + running-set
  + chunked-prefill + token budget per step. Attention stays in
  Apple code. Port only the scheduler layer when upstream Swift
  lands `BatchGenerator`.

### 2. Tiered KV cache

- **Files:** `omlx/cache/paged_cache.py`, `paged_ssd_cache.py`,
  `prefix_cache.py`, `tiered_manager.py`, `boundary_snapshot_store.py`,
  `memory_monitor.py`.
- **Structures:** `CacheBlock(block_id, ref_count, block_hash,
  doubly-linked-free-list-ptrs)`, `FreeKVCacheBlockQueue`,
  `BlockHashToBlockMap`, `PagedCacheManager(block_size=256)`,
  `PagedSSDCacheManager` with `PagedSSDCacheIndex` (OrderedDict LRU
  keyed on sha256).
- **Hashing:** `compute_block_hash` at `paged_cache.py:78` ‚Äî
  sha256 of `(model_name || parent_hash || token_ids || extra_keys)`.
  vLLM-style chained hash, longest-common-prefix match at block
  granularity (256 tokens). NOT exact-match-only.
- **Serialization:** hand-rolled `_write_safetensors_no_mx` (line 180)
  so background daemon thread (`ssd-cache-writer`) flushes without
  GPU sync. BF16 stored natively. 16 subdirs by first hex char.
- **Eviction:** LRU both tiers. Hot: `FreeKVCacheBlockQueue` pops
  oldest free. Cold: `evict_until_size(target)`. Default cold cap
  **100 GB**.
- **Tiering policy:** `TieredCacheManager` + `MemoryMonitor` ‚Äî
  under GPU memory pressure, blocks promoted/demoted.
- **Portable concepts for macMLX:** vLLM chained block hash, 256-token
  blocks, background writer thread avoiding MLX sync, two-level LRU,
  16-way sharded disk layout.

### 3. Multi-model load

- **File:** `omlx/engine_pool.py` ‚Äî `EnginePool`, `EngineEntry`.
- **Structure:** `dict[model_id ‚Üí EngineEntry]` + `asyncio.Lock` +
  `_max_model_memory`, `_current_model_memory`. Entry carries
  `estimated_size` (safetensors pre-scan), `last_access`, `is_pinned`,
  `is_loading`, `abort_loading`.
- **Policy:** pre-load memory check ‚Üí sort candidates by `last_access`
  ‚Üí evict LRU until fit ‚Üí load (serial, one lock). Pinned never
  evicted. Idle-TTL sweeper separately. `process_memory_enforcer`
  can abort in-progress loads.
- **Portable:** straightforward Swift-actor port.

### 4. MCP

- **Files:** `omlx/mcp/client.py`, `manager.py`, `executor.py`,
  `tools.py`, `config.py`, `types.py`. Optional dep: `mcp>=1.0.0`
  (official Python SDK).
- **Role:** MCP **client** only. `MCPClientManager` connects to
  configured upstream servers over stdio / SSE / streamable-HTTP in
  parallel. `MCPExecutor.execute_tool_calls` runs model-emitted tool
  calls in parallel/serial (semaphore), converts via
  `mcp_tools_to_openai` / `openai_call_to_mcp`.
- **Integration:** hooks into chat completions ‚Äî when model emits
  `tool_calls`, executor dispatches to MCP servers, feeds results
  back into next generation step. Same mental model as OpenAI
  function calling, transport-agnostic.
- **Portable:** stdio trivial (Process + pipes), streamable-HTTP is
  plain HTTP client. Borrow the OpenAI ‚Üî MCP tool-schema adapter
  verbatim (logic, not code ‚Äî different licenses).

### Architectural takeaways for macMLX

- **Copy the block-hash design verbatim.** vLLM-style chained sha256
  + 256-token blocks is industry standard.
- **Don't reinvent batching.** oMLX patches `mlx-lm.BatchGenerator`,
  doesn't replace it. Do the same with Swift when it lands.
- **Background SSD writer matters.** Hand-rolled safetensors avoids
  GPU-sync stalls on flush.
- **Two clean axes of eviction.** Block-level LRU (inside KV cache)
  fully separate from model-level LRU (inside engine pool).
- **MCP-as-client is the easy win.** No server runtime needed;
  spawn stdio MCP subprocesses.

**Risks to watch:** oMLX's tiered cache is paged-SSD-only ‚Äî when SSD
fills, everything pauses. Budget disk-pressure handling. Their
monkey-patch of `GenerationBatch._step` is fragile across mlx-lm
versions (pinned commit `dcbf6e3`) ‚Äî Swift port should extend, not
patch.

### Recommended SDK

`modelcontextprotocol/swift-sdk` (URL:
https://github.com/modelcontextprotocol/swift-sdk, pin `from: "0.11.0"`).
- Swift 6.0+, strict concurrency enabled
- Implements the 2025-11-25 MCP spec
- Both client and server roles
- stdio + streamable-HTTP + SSE
- Clean deps: swift-system, swift-log, swift-nio, mattt/eventsource
- MIT transitioning to Apache-2.0 ‚Äî compatible with our licensing
- Supports tools, resources, prompts, sampling, elicitation, roots,
  logging, cancellation, progress, OAuth

**Caveat:** pre-1.0 (0.11.x). Pin minor version. Wrap behind a thin
`MCPBridge` type.

### Integration surface

**Server role (primary, v0.4.0):**
- Tools: `chat_completion`, `list_models`, `load_model`,
  `unload_model`, `benchmark_model`, `open_in_app`
- Transport: stdio primary (Claude Desktop / Cursor spawn macmlx as
  subprocess); HTTP on `localhost:8000/mcp` secondary.
- Implementation: new CLI subcommand `macmlx mcp serve` ‚Üí `Server` +
  `StdioTransport` + tool registrations.

**Client role (v0.5.1+):**
- Settings UI registering external servers
- `~/.mac-mlx/mcp.json` (mirrors `claude_desktop_config.json`)
- Wire discovered tools into chat completions as OpenAI-style `tools[]`,
  route `tool_calls` via `Client.callTool(...)`.

### MVP for "macMLX has MCP"

One CLI subcommand `macmlx mcp serve` (stdio), two tools:
`list_models` + `chat`. ~200 LOC, one new file in CLI target, one
new module in `MacMLXCore/MCP/`. Claude Desktop / Cursor add
macmlx to `mcpServers` ‚Äî immediately runs local MLX inference
through it.

### Risks

- Pre-1.0 SDK: API churn. Mitigate with `MCPBridge` facade.
- SDK floor is macOS 13; we're on 14. Safe, re-verify on SDK bump.
- License transition MIT‚ÜíApache-2.0 is in flight; confirm terms on
  SPM dep add, record in `LICENSES.third_party`.
