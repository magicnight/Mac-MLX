# v0.3 — Vision-Language Model support (#23)

> Research + implementation plan. Research done against mlx-swift-lm
> 3.31.3 sources (already pinned in `MacMLXCore/Package.swift`).

## Research summary

### mlx-swift-lm's MLXVLM library is ready to import

The same SPM package we already depend on for `MLXLLM` ships
`MLXVLM` as a sibling product — no new dependency required. Located
at `MacMLXCore/.build/checkouts/mlx-swift-lm/Libraries/MLXVLM/` with:

- `VLMModel.swift` — protocol `VLMModel: LanguageModel, LoRAModel`
- `VLMModelFactory.swift` — registry of supported architectures
- `MediaProcessing.swift` — image resize / normalize / sRGB conversion
- `Models/` — concrete implementations (16 architectures, see below)

### Turn-key public API

```swift
import MLXVLM
import MLXLMHuggingFace
import MLXLMTokenizers

let model = try await loadModel(
    using: TokenizersLoader(),
    id: "mlx-community/Qwen2.5-VL-3B-Instruct-4bit"
)
let session = ChatSession(model)

let answer = try await session.respond(
    to: "what kind of creature is in the picture?",
    image: .url(URL(fileURLWithPath: "path/to/image.jpg"))
)
```

`ChatSession(_:)` is the same type we already use for text-only models
via `MLXLLM`. The `respond(to:image:)` overload accepts an `image:
UserInput.Image` (URL, `CGImage`, or `MLXArray` pixels).

### Supported architectures (out of the box)

| Model family | Example HF IDs |
|---|---|
| qwen2_vl / qwen2_5_vl / qwen3_vl | `mlx-community/Qwen2.5-VL-3B-Instruct-4bit`, `lmstudio-community/Qwen3-VL-4B-Instruct-MLX-4bit` |
| gemma3 | `mlx-community/gemma-3-4b-it-qat-4bit`, `…-12b-…`, `…-27b-…` |
| smolvlm / smolvlm2 | `mlx-community/SmolVLM-Instruct-4bit`, `HuggingFaceTB/SmolVLM2-500M-Video-Instruct-mlx` |
| paligemma | `mlx-community/paligemma-3b-mix-448-8bit` |
| mistral3 / pixtral / idefics3 / glm_ocr / fast_vlm / lfm2_vl | see `Libraries/MLXVLM/Models/` |

### What's **not** in mlx-swift-lm (we own or defer)

- **Image attachment UI** — NSImage picker, drag-drop, paste-from-clipboard
- **On-disk message persistence of images** — `ConversationStore.StoredMessage` currently holds only `content: String`
- **OpenAI `content` array format** — OpenAI's multimodal API shape is `content: [{ "type": "text", "text": "…" }, { "type": "image_url", … }]`. Our `/v1/chat/completions` route would need to parse this for external tools to submit images
- **Model detection** — `ModelLibraryManager` currently assumes every `.safetensors` dir is an LLM. VLMs have a `preprocessor_config.json` too; need to branch on `config.json`'s `model_type` or detect image processor presence

## Architecture

### 1. Data model (Core)

```swift
// ChatMessage — gain image attachments, backward-compatible
public struct ChatMessage: Codable, Hashable, Identifiable, Sendable {
    public let id: UUID
    public let role: MessageRole
    public let content: String
    public let images: [ImageAttachment]  // empty = text-only
    // ...
}

public struct ImageAttachment: Codable, Hashable, Sendable {
    /// File URL on disk (copied into `~/.mac-mlx/conversations/<uuid>/images/`
    /// on message save so the image survives the user moving the source).
    public let fileURL: URL
    /// Best-effort MIME type for OpenAI-API payload construction.
    public let mimeType: String
}
```

Backward compat: JSON decoder default-constructs `images: []` when the
field is absent. Existing v0.2 conversations round-trip unchanged.

### 2. LocalModel — tag VLMs separately

```swift
public enum ModelFormat: String, Codable, Sendable {
    case mlx          // text-only
    case mlxVLM       // vision-language (new)
    case gguf, unknown
}
```

`ModelLibraryManager.scan(_:)` inspects `config.json`'s `model_type`:
known VLM types (`qwen2_vl`, `qwen3_vl`, `gemma3`, `smolvlm`, etc.)
→ `.mlxVLM`. Everything else stays `.mlx`.

Compatibility UI: VLM models get a small 🖼 badge in the library; the
chat input only offers the image picker when `currentModel.format == .mlxVLM`.

### 3. Engine — branch at load time

`MLXSwiftEngine.load(_:)` inspects `model.format`:

- `.mlx` → existing `MLXLLM.loadModelContainer(from:)` path, unchanged
- `.mlxVLM` → new `MLXVLM.loadModel(using:id:)` path that returns a
  `VLMModel`-wrapping `ModelContainer`

Cache the loaded container in a new `private var modelSupport: ModelSupport` enum:

```swift
private enum ModelSupport {
    case llm(ModelContainer)
    case vlm(VLMContainer)
    case none
}
```

`generate(_:)` switches on `modelSupport`:
- `.llm` → existing flow, ignores any `ImageAttachment` on the last message (emit warning to Pulse)
- `.vlm` → build `UserInput` including images, use `ChatSession(vlm).respond(to:image:)`

### 4. UI — ChatInputView + ChatMessageView

- **ChatInputView**: add `photo.on.rectangle` toolbar button next to Send.
  Opens `NSOpenPanel` with image-only file types. Selected images appear
  as thumbnails above the text field. Shift+Drag to reorder; click-× to
  remove. Cmd-V pastes clipboard image.
- **ChatMessageView**: if `message.images.isNotEmpty`, render inline
  `SwiftUI.Image` thumbnails (max 120pt square) before the text content,
  clickable to open in Preview via `NSWorkspace.shared.open(_:)`.
- **Current-model gate**: image picker disabled + tooltip
  *"Load a vision-capable model to attach images"* when
  `coordinator.currentModel?.format != .mlxVLM`.

### 5. OpenAI-compat HTTP (HummingbirdServer)

OpenAI API shape for multimodal:

```json
{
  "model": "Qwen2.5-VL-3B-Instruct-4bit",
  "messages": [
    { "role": "user", "content": [
      { "type": "text", "text": "What's in this image?" },
      { "type": "image_url", "image_url": { "url": "data:image/png;base64,AAA…" } }
    ]}
  ]
}
```

`HummingbirdServer` currently decodes `content: String`. Extend to accept
both:
- String → existing behaviour
- Array of parts → extract text parts into `content`, decode base64
  images into temp files and attach.

Data-URL + http(s) URL parsing; reject `file://` (security — server is
LAN-reachable via localhost only by default, but still). Max 10 MB per
image, max 4 images per message.

### 6. Persistence — copy images to conversation dir

On `ConversationStore.save(_:)`:
- For each `ImageAttachment`, if `fileURL` is outside
  `~/.mac-mlx/conversations/<conv-uuid>/images/`, copy it there and
  rewrite the attachment's URL. This makes the conversation portable
  (user can move their model directory without images going missing)
  and deterministic under App Sandbox.
- On `delete(id:)`, remove the conversation's `images/` subdirectory.

### 7. Benchmark — defer

`BenchmarkRunner`'s generate callback passes a text-only
`GenerateRequest`. VLM bench would need an image corpus + per-image
timing. Skip for v0.3; file follow-up issue.

## Build order

1. `MacMLXCore`: `ChatMessage.images` + `ImageAttachment` + JSON backcompat — tests for decode of v0.2 JSON without `images` field.
2. `MacMLXCore`: `ModelFormat.mlxVLM` + `ModelLibraryManager` detection — tests via fixture `config.json` files.
3. `MacMLXCore`: `MLXSwiftEngine` VLM branch + `ModelSupport` enum.
4. `macMLX`: ChatInputView image picker + thumbnails, ChatMessageView inline render.
5. `MacMLXCore`: `HummingbirdServer` multimodal `content` array parsing + tests.
6. `MacMLXCore`: `ConversationStore.save` image-copy step + `delete` cleanup.
7. `macMLX`: UI gate on `format == .mlxVLM`.

Ship as `v0.4.0` (feature-size). Budget: ~1-2 weeks of focused work.

## Exit criteria

- User can download `mlx-community/Qwen2.5-VL-3B-Instruct-4bit` via
  the Models tab, load it, attach a JPEG via the image picker, and
  get a coherent response.
- OpenAI-compat endpoint accepts the multimodal `content` array from
  at least one external client (curl test fixture).
- Conversation containing images round-trips through save/reload.
- 60 + current Core tests remain green; >= 10 new tests cover the
  data model, detection, and HTTP parsing.
- Xcode Release build green; CLI build green.

## Open questions / risks

1. **Sandbox image access**: The user-selected-file entitlement already
   covers NSOpenPanel results. Drag-drop onto the chat view needs the
   same security-scoped access — worth verifying with a paste/drop
   test on a sandboxed dev build.
2. **Memory footprint**: VLMs load both text + vision towers. Some of
   the listed 12B/27B Gemma-3 variants are >15 GB. Our compatibility
   check (`MemoryProbe.totalMemoryGB`) already warns; maybe tighten
   the threshold for VLMs.
3. **Tokenizer `applyChatTemplate` for multimodal**: Our
   `TokenizerBridge` in `MLXSwiftEngine.swift` handles text messages.
   VLM chat templates often inject `<image>` tokens at specific
   positions. `MLXVLM`'s factory wires this up via the `UserInputProcessor`
   protocol; we just need to make sure our bridge doesn't override.
4. **Video support**: `SmolVLM2-Video` and some Qwen variants accept
   video. Out of scope for v0.4 (defer to v0.5+).
