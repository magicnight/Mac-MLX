### Branch / worktree layout

| Sub-plan file | Branch | Module | Touches |

### Why these splits don't conflict

- **Source files**: every module owns disjoint files. No edit overlap.
- **`Package.swift`**: each subagent uncomments **its own dep line** in the `dependencies:` array of the `MacMLXCore` target. Merge order matters â€” see Phase 2.
- **Tests**: each test target is per-module file, no shared test fixtures except `StubInferenceEngine` (which only the server uses, read-only).

## Phase 2 â€” Sequential Merge to `main`

Subagent worktrees finish independently. Main session merges in this order to minimise `Package.swift` conflicts:

1. **`feat/stage3-settings`** first â†’ adds Pulse product line (new line, no overlap)
2. **`feat/stage3-engine`** â†’ uncomments MLXLLM product line (existing commented line)
3. **`feat/stage3-server`** â†’ uncomments Hummingbird product line (existing commented line)
4. **`feat/stage3-library`** last â†’ only edits Sources, no `Package.swift` touch (uses URLSession from Foundation only)

Each merge:
```bash
git checkout main
git merge --no-ff feat/stage3-<module> -m "feat(stage3): merge <module>"
swift test --package-path MacMLXCore     # verify nothing regressed
git push origin main
```

If a merge conflicts on `Package.swift`: resolve manually by accepting both adjacent product entries (they're separate lines in the array).

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| `MLXLLM` API differs from spec snippet (spec is illustrative) | Engine sub-plan opens with a "verify API by reading mlx-swift-lm source" task before writing code |
| Subagent goes off the rails in worktree | Each subagent returns commit SHA + test output; main session reviews before merging. Worktree auto-cleaned by Agent tool |
| `Package.swift` merge conflict | Phase 2 merge order is deterministic; conflicts (if any) are trivial to resolve (separate lines) |
| End-to-end test fails in CI due to network/disk | Marked `.disabled` for CI; runs only locally on demand |
| Pulse Logger API doesn't match my reading | Settings sub-plan opens with a "verify Pulse 5.x ConsoleView/storeMessage signatures" task |
| Subagents take vastly different times | Parallelism gain is bounded by the slowest (likely Engine due to MLX integration). Worst case: longest finishes alone, others sit done â€” same as sequential timing on the slow path. No downside. |

## Self-Review

**Spec coverage:** All four Stage 3 modules from `v0.1-overview.md` have an assigned worktree with a sub-plan to be written. Phase 0 covers the cross-cutting Package.swift edits the overview didn't anticipate.

**Decomposition rationale:** Modules grouped by data ownership boundary, not by technical layer. Each subagent owns one cohesive piece of functionality and its tests, with no shared mutable state.

**Known gaps to address in sub-plans:**
- MLXLLM exact API surface â€” engine sub-plan must verify before coding
- Pulse 5.x Logger API â€” settings sub-plan must verify before coding
- Hummingbird 2.x routing API â€” server sub-plan must verify before coding (was 1.x in many older docs)

These three "verify before code" preflight steps are the single biggest risk reduction. Without them, any of the three subagents could hallucinate APIs.
